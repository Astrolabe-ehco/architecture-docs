# 风险与技术债务 (Risks and Technical Debt)

## 业务与技术风险
* **宿舍物理机断电与断网:** 服务端部署于宿舍物理机之上，面临宿舍夜间断电、校园网故障或内网穿透节点波动的影响，这会导致客户端上报失败或服务失联。
  * *缓解措施:* 在移动端（如鸿蒙 App）增加强大的**离线缓存与补偿机制**（如本地 SQLite 缓存储数据），在网络和宿主机恢复后执行断点续传；考虑为物理机配置软路由自启动及定时开机策略。
* **内网穿透节点的带宽瓶颈:** 外部 NAT 穿透服务（无论是自行搭建的 FRP 还是第三方公网隧道）的公网带宽通常较小。
  * *缓解措施:* 保持客户端与服务端的数据交互以纯文本/JSON 的微量请求为主，避免不必要的大文件传输。
* **鸿蒙后台保活限制:** HarmonyOS 对非音乐/导航类应用的后台驻留有极其严格的管控。如果番茄钟在后台被系统强杀，将导致时间切片记录中断。
  * *缓解措施:* 申请特权接口，或采用推送唤醒/卡片刷新等机制变相保活；深度结合手表端运动传感器作为辅助判断。
* **大模型幻觉 (Hallucination):** AI 生成的“神谕”可能会出现逻辑断裂、甚至给出有害的生活建议。
  * *缓解措施:* 在 Go 后端调用大模型时引入强有力的 Prompt 工程约束，要求大语言模型遵循严格的模板进行输出，并增加必要的容错和重试机制。

## 技术债务
* 系统作为单体服务，目前严重依赖单一关系型数据库节点。若数据量随时间不断增长，可能需要通过建立历史数据归档机制或引入本地轻便的数据检索方案来缓解单点存储压力。
