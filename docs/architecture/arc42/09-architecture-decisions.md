# 架构决策记录 (Architecture Decisions)

## [ADR-001] 采用 Go 语言构建基础级单体后端 (Monolithic Architecture)
* **背景:** Astrolabe 系统主要是针对个人的真实时间量化记录，作为单用户工具，其预期 QPS 极低，完全没有任何高并发分布式协同或海量访问负载处理的要求。
* **决策:** 统一采用 Go 语言（基于 Gin 框架）构建单一整体后端服务，将数据接收处理以及调用外部 LLM API 等所有核心逻辑集中在一个应用实例中闭环完成。
* **后果:** 降低了系统复杂性与部署成本，所有的 API 和数据都在一个应用上下文中流通，极大地方便了后期的维护与服务器部署操作。

## [ADR-002] 客户端打点直接同步落入关系型数据库
* **背景:** 个人使用场景下单次请求体量较小且频次相对规律稳定，不需要通过额外的缓冲层或队列缓存手段来对抗瞬间的大流量写入冲击。
* **决策:** 移动客户端、手表端产生的数据行为与体征数据均由 Go 后端单体同步通过 SQL `INSERT` 执行落盘直接写入稳定的关系型数据库（如 SQLite / MySQL）。
* **后果:** 保障了基础架构组件开销到达最小化，并实现了单次记录入库的强原子性与可靠性，彻底杜绝了因内存缓冲问题导致的秒级丢失风险。
